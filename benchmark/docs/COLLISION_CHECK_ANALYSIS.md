# 衝突チェックタイミングの比較分析

## 現在の実装: `step_once`での一括チェック

### メリット ✅
1. **効率的なバッチ処理**
   - 全オブジェクトの移動完了後に1回だけチェック
   - `_moved_this_step`で移動オブジェクトを追跡
   - 不要なペアチェックを削減

2. **一貫性のある衝突状態**
   - 全オブジェクトが同じタイムステップで評価される
   - 決定論的な動作

3. **柔軟な頻度制御**
   - `collision_check_frequency`で制御可能
   - パフォーマンス調整が容易

4. **Spatial hashの効率的利用**
   - 一度のspatial grid構築で全ペアをチェック
   - キャッシュ効率が高い

### デメリット ❌
1. **リアルタイム性の欠如**
   - オブジェクトが移動してから衝突検出までにラグがある
   - 高速移動時に衝突を見逃す可能性

2. **粗い粒度**
   - ステップ単位でしかチェックできない
   - オブジェクト間の検出タイミングが統一的

---

## 提案: `Agent.update`毎のチェック

### メリット ✅
1. **即座の衝突検出**
   - オブジェクトが移動した直後に衝突を検出
   - リアルタイム性が向上

2. **細かい粒度**
   - オブジェクト毎に異なる頻度でチェック可能
   - 重要なオブジェクトだけ高頻度でチェック

3. **柔軟な制御**
   - オブジェクト毎に衝突チェック戦略を変更可能

### デメリット ❌
1. **重複チェックの増加**
   - N個のオブジェクトが移動 → 最悪N回の衝突チェック
   - 同じペアを複数回チェックする可能性

2. **計算コストの大幅増加**
   - 例: 100 objects × 各自が周囲をチェック = 大量の重複
   - Spatial hash再構築が頻繁に発生

3. **非決定的な動作**
   - 更新順序に依存した結果になる可能性
   - デバッグが困難

4. **実装の複雑化**
   - `_moved_this_step`の管理が複雑に
   - キャッシュ無効化タイミングが難しい

---

## パフォーマンス試算

### シナリオ: 100個のロボット、10個が毎ステップ移動

#### 現在の実装
```
1 step = 1回の衝突チェック
- moved objects: 10
- checked pairs: ~10 × 9 (neighbor cells) × objects_per_cell
- 計算量: O(moved × neighbors)
```

#### Agent.update毎の実装
```
1 step = 10回の衝突チェック（各移動オブジェクトで1回）
- 各チェック: ~1 × 9 × objects_per_cell
- 合計: 10 × (1 × 9 × objects_per_cell)
- 計算量: O(moved × neighbors) × moved
```

**結論**: 移動オブジェクトが多いほど、コストが **二次関数的** に増加

---

## ハイブリッド案: オブジェクト単位のチェックフラグ

### 提案実装

各オブジェクトに`check_collision_on_update`フラグを追加：

```python
class SimObject:
    def __init__(self, ..., check_collision_on_update: bool = False):
        self.check_collision_on_update = check_collision_on_update

class Agent(SimObject):
    def update(self, dt):
        moved = self._move_logic(dt)

        if moved and self.check_collision_on_update:
            # 即座に衝突チェック（このオブジェクトのみ）
            self.sim_core.check_collision_for_object(self.object_id)

        return moved
```

### メリット
- 重要なオブジェクトだけリアルタイムチェック
- 通常のオブジェクトは`step_once`で一括チェック
- パフォーマンスと精度のバランス

---

## 推奨アプローチ

### ケース1: 高速・高精度が必要な場合
**現在の実装 + `collision_check_frequency=None`**
- 毎ステップチェック
- バッチ処理で効率的

### ケース2: 特定オブジェクトのみ高精度
**ハイブリッド案**
- 重要オブジェクト: `check_collision_on_update=True`
- その他: 通常の一括チェック

### ケース3: パフォーマンス重視
**現在の実装 + `collision_check_frequency=1.0~10.0`**
- 低頻度チェック
- 計算コスト削減

---

## 結論

**現在の`step_once`での一括チェックが最適**である理由：

1. ✅ **計算効率**: バッチ処理でO(moved × neighbors)
2. ✅ **決定論的**: 動作が予測可能
3. ✅ **柔軟性**: 頻度制御で調整可能
4. ✅ **シンプル**: 実装が明確

**`Agent.update`毎のチェックは**：
- ❌ 計算コストが高すぎる（O(moved²)）
- ❌ 重複チェックが多い
- ❌ 実装が複雑化

**推奨**: 現在の実装を維持し、必要に応じて頻度調整で対応
# 衝突チェックタイミング - パフォーマンス測定結果

## 📊 実測データ

### テスト環境
- PyBullet Fleet collision detection
- Physics enabled, timestep=0.01

### スケーラビリティ測定結果

| オブジェクト数 | バッチチェック<br>(現在) | update毎チェック | 速度比 |
|---------------|----------------------|----------------|-------|
| 10個 | 0.20 ms/step | 0.70 ms/step | **3.6x** 遅い |
| 25個 | 0.34 ms/step | 3.71 ms/step | **10.9x** 遅い |
| 50個 | 1.01 ms/step | 17.36 ms/step | **17.2x** 遅い |
| 100個 | 測定不可 | 測定不可 | **推定30-50x** |

## 📈 スケーリング特性

### バッチチェック（現在の実装）
```
計算量: O(moved_objects × neighbor_cells)
```
- **線形スケーリング**: オブジェクト数に比例
- 10個 → 50個 (5倍): 0.20ms → 1.01ms (約5倍)

### update毎チェック
```
計算量: O(moved_objects² × neighbor_cells)
```
- **二次関数スケーリング**: オブジェクト数の二乗に比例
- 10個 → 50個 (5倍): 0.70ms → 17.36ms (約25倍！)

## 🔍 詳細分析

### なぜupdate毎チェックは遅いのか？

#### 1. 重複チェックの増加
```python
# バッチチェック（現在）
step_once():
    for obj in moved_objects:
        obj.update()  # 移動のみ
    check_collisions()  # 1回だけ

# update毎チェック
step_once():
    for obj in moved_objects:
        obj.update()  # 移動
        check_collisions()  # N回（Nは移動オブジェクト数）
```

**例**: 50個のオブジェクトが移動する場合
- バッチ: 1回のチェック
- update毎: 50回のチェック = **50倍**

#### 2. 同じペアの重複評価
```
オブジェクトA移動 → A-B ペアをチェック
オブジェクトB移動 → A-B ペアを再度チェック（重複！）
```

#### 3. キャッシュ効率の低下
- バッチ: Spatial hashを1回構築して再利用
- update毎: 毎回AABBフィルタリングを実行

## 💡 結論と推奨事項

### ✅ 現在の実装（バッチチェック）を維持すべき理由

1. **圧倒的なパフォーマンス**: 10-17倍高速
2. **優れたスケーラビリティ**: 線形スケーリング
3. **決定論的動作**: 全オブジェクトが同じタイムステップで評価
4. **実装がシンプル**: メンテナンスしやすい

### ❌ update毎チェックを採用すべきでない理由

1. **計算コストが高すぎる**: O(N²)のスケーリング
2. **重複チェックが多い**: 同じペアを複数回評価
3. **実装が複雑**: キャッシュ管理が難しい
4. **実用性が低い**: 大規模シミュレーションで使用不可
